# Концепция: онлайн-тренажёр языка 1С (BSL) с синтакс-помощником

> **Статус:** концепт / обсуждение (ещё не реализуется)
> **Дата:** 2026-06-11
> **Версия:** 0.2
> **Назначение:** браузерный тренажёр для обучения и тренировки базовому синтаксису встроенного языка 1С (BSL) — с пошаговой отладкой и продвинутым синтакс-помощником. В перспективе — встраивание в LMS.

---

## 1. Что это и зачем

Браузерный инструмент, в котором можно **писать код на BSL, исполнять его и отлаживать пошагово**, не разворачивая платформу 1С. Цель — обучение и тренировка базового синтаксиса языка.

Сознательно **не реализуем** платформенный слой: метаданные (справочники, документы, регистры), формы, криптографию, файлы, HTTP, внешние компоненты и т.п. Это тренажёр **языка**, а не эмулятор платформы.

Поверх тренажёра — **синтакс-помощник (СП)** уровня лучших современных IDE и doc-сайтов, заметно превосходящий справку конфигуратора.

---

## 2. Методика (reading-first / PRIMM)

Тренажёр — не просто «среда чтобы писать». Его педагогическая ценность в том, что он растворяет ложную дихотомию «сначала читать или сначала писать» и делает обе стороны одним непрерывным процессом.

### 2.1 Ложная дихотомия «читать vs писать»

Распространённый тезис «чтобы научиться — нужно писать» верен не про письмо как таковое, а про **тесную петлю обратной связи**. Письмо без наблюдения за результатом — это барахтанье: новичка сажают перед пустым листом, он получает невнятную ошибку и не выстраивает причинно-следственную связь. Работает не «много писать», а «писать и сразу видеть, что именно произошло».

Установка «сначала читать» (основа курса про чтение кода) и требование «нужно писать» не противоречат друг другу — между ними есть пропущенная ступень, которую большинство курсов проскакивает: **активное чтение через предсказание и трассировку**.

### 2.2 Отладчик как усилитель чтения

Пошаговый отладчик по своей природе — инструмент **чтения**, а не письма. Идти по коду, смотреть, как меняются переменные, как переменная меняет тип, что лежит в `Структуре` — это допрашивающее, активное чтение, а не пассивное проглядывание. Тренажёр делает чтение **исполняемым**: ученик не просто смотрит на код, а наблюдает его поведение.

То есть инструмент обслуживает установку «сначала читать», а не отменяет её — и одновременно даёт ту самую петлю обратной связи, которой не хватает «голому» письму.

### 2.3 PRIMM как каркас

Эта интуиция совпадает с реальной методикой из CS-образования — **PRIMM** (Predict–Run–Investigate–Modify–Make, Сью Сентанс), построенной ровно на принципе «сначала работаем с чужим кодом (читаем), и только потом пишем своё». Маппинг на тренажёр почти буквальный:

| Этап PRIMM | В тренажёре |
|---|---|
| **Predict** — предскажи результат | «Что выведет этот код?» до запуска |
| **Run** — запусти | Пошаговое исполнение |
| **Investigate** — разберись | Инспекция переменных, типов, стека (живая визуализация = Investigate, сделанный осязаемым) |
| **Modify** — измени | Поменять runnable-пример и перезапустить |
| **Make** — создай | Написать с нуля в той же среде |

### 2.4 Что это даёт курсу

Курс про чтение кода — это **передняя половина** PRIMM (Predict / Run / Investigate). Тренажёр естественно дотягивает до **Modify / Make**. По сути это **PRIMM-движок для 1С**, где чтение и письмо живут в одном инструменте, а не в разных мирах.

Конкретный формат заданий, прямо обслуживающий reading-first: **«предскажи вывод → запусти → сравни»**. Ученик записывает ожидаемый результат, тренажёр показывает дифф между предсказанием и фактом. Это спиральные задания курса, но с автопроверкой **предсказания**, а не только итогового кода — то есть проверяется именно понимание, а не способность подобрать рабочий код наугад.

## 3. Принципиальная осуществимость

**Реально.** С точки зрения теории языков BSL скромен: процедурный, динамически типизированный, без пользовательских классов и замыканий, с простой объектной моделью. Tree-walking интерпретатор такого языка — стандартная, хорошо изученная задача. Браузер не помеха: всё считается на клиенте в JS/TS.

Реалистичная оценка MVP (синтаксис + core-типы + набор глобальных функций + пошаговый отладчик + редактор) — **недели сосредоточенной работы**, не выходные и не год.

---

## 4. Скоуп MVP

**Решение:** только чистый синтаксис + core-типы. Всё остальное — потом.

### В скоупе

**Синтаксис языка:**
- Объявления: `Перем`, `Процедура`/`Функция` (с `Экспорт`, `Знач`, параметрами по умолчанию)
- Управление потоком: `Если/ИначеЕсли/Иначе/КонецЕсли`, `Для..По..Цикл`, `Для Каждого..Из..Цикл`, `Пока..Цикл`, `Прервать`, `Продолжить`, `Возврат`
- Исключения: `Попытка..Исключение..КонецПопытки`, `ВызватьИсключение`
- Операторы: арифметика, сравнение, `И`/`ИЛИ`/`НЕ`, тернарный `?()`, конкатенация
- Литералы: числа, строки (многострочные `|`, экранирование `""`), даты, `Истина`/`Ложь`, `Неопределено`, `Null`
- Конструктор `Новый`, вызовы методов и обращение к свойствам через точку

**Core-типы:**
- Примитивы: `Число`, `Строка`, `Дата`, `Булево`, `Неопределено`, `Null`, `Тип`
- Коллекции: `Массив`, `Структура`, `Соответствие`, `СписокЗначений`, `ТаблицаЗначений`
- Глобальные функции: строковые, числовые, даты, преобразования, `ТипЗнч`, `Сообщить`

> `ТаблицаЗначений` — единственный «тяжёлый» из простых типов (колонки, строки, итерация, поиск). Делать **последней** в очереди типов: на ней проверяется, тянет ли архитектура реальную объектную модель.

### Отложено (явно вне MVP)

Язык запросов · метаданные (справочники/документы/регистры) · формы · граница Клиент/Сервер · платформенные объекты (крипто, файлы, HTTP, XML/JSON) · внешние компоненты.

### Учебные акценты, доступные уже в MVP

Эти вещи влезают даже в «чистый синтаксис» и именно на них спотыкаются новички — а тренажёр, в отличие от настоящей 1С, может их **визуализировать**:
- переменная, **меняющая тип в рантайме** (живая инспекция типа);
- разница **передачи коллекций по ссылке против `Знач`**.

## 5. Архитектура

### 5.1 Что переиспользуем (open-source экосистема BSL)

| Кирпич | Что это | Как используем |
|---|---|---|
| **Monaco** | Редактор кода из VS Code, работает в браузере | Редактор + база для автодополнения/hover/сигнатур |
| **Грамматика 1c-syntax (TextMate)** | Подсветка BSL | Берём готовой |
| **ANTLR-грамматика BSL** (1c-syntax) | Генерирует парсер, в т.ч. на **TypeScript** | AST в браузере почти даром |
| **OneScript** (EvilBeaver) | Полный открытый интерпретатор BSL на C#, парсер `OneScript.Language`, отладчик по DAP | Эталон поведения; источник машиночитаемых сигнатур глобального контекста |
| **bsl-language-server** (1c-syntax) | LSP на Java: диагностика, символы, фолдинг | Источник метаданных типов/методов; референс по фичам |

Ссылки: `github.com/EvilBeaver/OneScript` · `github.com/1c-syntax/bsl-language-server` · `github.com/1c-syntax` (грамматики).

### 5.2 Рекомендуемая схема: чужой парсер, свой интерпретатор

```
Monaco + готовая подсветка
        │
        ▼
ANTLR-грамматика BSL → TS-парсер → AST
        │
        ▼
Собственный tree-walking интерпретатор (TS) с отладчиком
```

**Почему свой интерпретатор:** парсер — самая нудная часть, её берём готовой; исполнитель — самая ценная, и владеть им надо целиком, потому что **отладка почти бесплатна только тогда, когда сам управляешь циклом вычисления**. Точка останова = «проверь breakpoint перед оператором», стек вызовов = собственный список фреймов, инспекция переменных = чтение текущего scope.

### 5.3 Каталог языка как единый источник правды

Ключевая идея: **СП — это не отдельная фича, а тот же каталог языка, отрендеренный несколькими способами.** Интерпретатору всё равно нужен реестр своих типов и функций. Делаем его не кодом-вразброс, а структурным каталогом (YAML/JSON):

```
тип / метод / функция →
    сигнатура, параметры (тип, опциональность),
    возвращаемое значение, описание,
    имена ru/en, категория, примеры
```

Из одного источника кормятся:
1. **Рантайм** интерпретатора (что реально исполняется);
2. **Monaco-провайдеры** — автодополнение, hover, signature help;
3. **Панель-справочник** (СП) — браузинг и поиск.

**Инварианты (механизм отдельно от политики):**
- Функция есть в рантайме, но не в каталоге (или наоборот) → **ошибка сборки**. Документация физически не может разойтись с поведением.
- Примеры из каталога **прогоняются тем же интерпретатором в CI** (executable docs / doctest) → не протухают.

## 6. Отладчик в браузере

Браузер однопоточный: наивный синхронный обходчик не умеет «встать на брейкпоинте и держать UI живым». Три чистых решения:

| Вариант | Суть | Плюсы | Минусы |
|---|---|---|---|
| **A. Генераторный интерпретатор** | `function*` + `yield` на каждом операторе; драйвер двигает по шагам, пауза = не звать `.next()` | Чистый main-thread, без спецзаголовков, шаг into/over/out естественно | Evaluator многословнее, ниже перформанс (для тренажёра не важно) |
| **B. Worker + SharedArrayBuffer + Atomics.wait** | Синхронный интерпретатор в воркере реально блокируется на брейкпоинте | Обычный синхронный код, честная блокировка | **Требует заголовков COOP/COEP** (cross-origin isolation) |
| **C. async/await на каждом шаге** | `await` чекпоинта на каждом операторе | Просто в голове | Overhead, протекающая асинхронность |

**Выбор для тренажёра: вариант A (генераторы).** Не требует особых заголовков, живёт в главном потоке, модель шага 1:1 ложится на то, что видит ученик. (См. §8 — на GitHub Pages вариант B вообще недоступен.)

**Педагогический бонус собственного исполнителя:** живая отрисовка стека и переменных, тип каждой переменной, визуализация содержимого `Структура`/`Соответствие`/`ТаблицаЗначений`, потенциально step-back — то, что настоящая 1С не показывает.

## 7. Синтакс-помощник (СП)

### 7.1 Чем бьём конфигуратор

Справка конфигуратора неплоха, но это статичное дерево. Берём лучшее из мира программирования:
- **Поиск по смыслу, а не только по дереву:** fuzzy + ru/en + по намерению («как обрезать пробелы» → `СокрЛП`). Дерево остаётся как один из режимов.
- **Runnable-примеры:** у каждой статьи кнопка «в тренажёр» → код открывается, запускается, **проходится отладчиком пошагово**. Уровень MDN «try it» / Rust Playground.
- **Inline-помощь LSP-грейда:** signature help (на каком параметре, типы, дефолты), hover с описанием и возвращаемым типом, go-to-definition по типам внутри примера.
- **Онлайн, шарится по URL, не привязан к десктопу.**

### 7.2 AI-слой (Фаза 2)

Вишенка, но с дисциплиной: **AI садится поверх каталога как заземление, а не вместо него.** Если дать модели генерить сигнатуры из головы — выдумает несуществующие методы, и СП перестанет быть источником правды.

- **Ядро СП — детерминированное и корректное** (каталог = ground truth).
- **AI делает то, что детерминизмом не закрыть:** NL-поиск, «объясни эту ошибку», «почему тут `Неопределено`», «сгенерируй пример под задачу» — с текущим кодом и каталогом в контексте.

## 8. Инфраструктура: GitHub + GitHub Pages

**Ядро (Фазы 0–1) — на 100% статично** → GitHub Pages идеально: бесплатно, версионируется, форкается, шарится по URL. Каталог в репозитории → новые функции и примеры принимаются через **PR** (открытая модель).

**Ключевое ограничение Pages:** нельзя ставить свои HTTP-заголовки → нет COOP/COEP → нет `SharedArrayBuffer` → **вариант отладчика B недоступен.** Это решающий аргумент за **генераторный интерпретатор (вариант A)**. (Костыль `coi-serviceworker` существует, но хрупкий — не закладываемся.)

**Где статика честно ломается (нужен тонкий бэкенд):**

| Потребность | Почему не статика | Решение |
|---|---|---|
| **AI-слой** (Фаза 2) | Ключ Anthropic нельзя в публичную статику — вытащат и сольют кредиты | Тонкий прокси, держащий ключ: **Coolify VPS** (уже планируется) или Cloudflare Worker (free). Либо режим «свой ключ» в браузере |
| **Прогресс / многопользовательность** (Фаза 3) | Pages не хранит per-user данные на сервере | MVP — `localStorage`/`IndexedDB` в браузере; аккаунты и оценки — **Coolify VPS** |

**Сборка:** GitHub Actions собирает статический бандл и деплоит на Pages, там же прогоняет примеры каталога как doctest и проверяет инвариант «рантайм ↔ каталог». Мелочь Pages-SPA — deep-linking лечится hash-роутингом или трюком с `404.html`.

**Итог:** всё ядро и весь СП — статично на Pages; сервер появляется только там, где принципиально нужен (AI и общий прогресс), и ровно под это уже стоит Coolify.

## 9. Дорожная карта по фазам

| Фаза | Содержание | Хостинг |
|---|---|---|
| **0** | Каталог языка (YAML) как единый источник правды (~10 типов + ~50–80 глобальных функций ≈ 100–150 записей). Засидить из открытых данных OneScript/bsl-language-server, где можно | GitHub Pages |
| **1** | Лексер/парсер (ANTLR) → AST → генераторный интерпретатор с пошаговой отладкой. Monaco + подсветка. Monaco-провайдеры из каталога + панель-справочник с поиском + runnable-примеры. «СП онлайн» на детерминированном ядре | GitHub Pages |
| **2** | AI-тьютор поверх каталога (NL-поиск, объяснение ошибок, генерация примеров) | Pages + прокси (Coolify / CF Worker) |
| **3** | Встраиваемый компонент с event-API → обёртка под LMS (LTI/xAPI/SCORM) | Pages + Coolify VPS |

> **Архитектурное решение, дешёвое сейчас и окупаемое потом:** строить тренажёр как **встраиваемый компонент с чистым event-API** (load code → run → события: вывод, попадание на брейкпоинт, ошибка, завершение), без допущения, что он живёт на отдельной странице. Тогда LMS-встраивание — это «просто» обёртка.

## 10. Особенности BSL для лексера/парсера

Что важно учесть, чтобы было «по-настоящему 1С»:
- **Двуязычные ключевые слова:** `Если`/`If`, `КонецЕсли`/`EndIf`, `Новый`/`New`, `Процедура`/`Procedure` и т.д.
- **Регистронезависимость** ключевых слов и идентификаторов.
- **Литералы дат:** `'20240101'`, `'20240101120000'`.
- **Строки:** экранирование `""`, многострочные с продолжением `|`.
- **Директивы препроцессора:** `#Если`, `#Область`/`#КонецОбласти`.
- **Директивы компиляции:** `&НаКлиенте`, `&НаСервере` — в MVP можно парсить и игнорировать (либо позже симулировать границу как учебный приём).
- **Передача параметров:** `Знач` (по значению) против передачи по ссылке.

## Глоссарий

- **BSL** — Built-in Language, встроенный язык 1С:Предприятие.
- **СП** — синтакс-помощник.
- **Tree-walking интерпретатор** — исполнитель, обходящий узлы AST и вычисляющий их напрямую.
- **AST** — абстрактное синтаксическое дерево.
- **LSP / DAP** — Language Server Protocol / Debug Adapter Protocol.
- **COOP/COEP** — заголовки cross-origin isolation, нужные для `SharedArrayBuffer`.
- **Каталог языка** — структурированное описание типов/методов/функций; единый источник правды для рантайма, СП и Monaco.
- **reading-first** — педагогический принцип «сначала чтение кода, потом письмо».
- **PRIMM** — методика обучения программированию (Predict–Run–Investigate–Modify–Make, Сью Сентанс): работа с чужим кодом перед написанием своего.
